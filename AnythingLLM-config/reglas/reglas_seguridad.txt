[REGLAS DE SEGURIDAD]

Estas reglas se aplican en TODAS las tareas, sin excepción.
La seguridad tiene prioridad sobre rendimiento y simplicidad.

---

[PHP — VULNERABILIDADES CRÍTICAS]

SQL INJECTION:
- Nunca concatenar variables en queries SQL
- Usar siempre PDO con prepared statements y parámetros enlazados
- Nunca: "SELECT * FROM users WHERE id = " . $id
- Siempre: $stmt = $pdo->prepare("SELECT * FROM users WHERE id = ?")

XSS (Cross-Site Scripting):
- Nunca hacer echo de datos del usuario sin escapar
- Usar htmlspecialchars($var, ENT_QUOTES, 'UTF-8') al renderizar en HTML
- Nunca confiar en $_GET, $_POST, $_COOKIE sin sanitizar

CSRF:
- Tokens CSRF en todos los formularios que modifican datos
- Validar el token en el servidor antes de procesar

EXPOSICIÓN DE ERRORES:
- display_errors = Off en producción
- Loggear errores internamente, nunca mostrarlos al usuario
- Nunca exponer rutas de archivos o stack traces

OTROS:
- Nunca hardcodear credenciales en código
- Variables de entorno para configs sensibles
- Validar y sanitizar toda entrada antes de procesar

---

[JAVASCRIPT — VULNERABILIDADES CRÍTICAS]

EVAL Y CÓDIGO DINÁMICO:
- Nunca usar eval(), Function(), setTimeout con string
- Nunca usar innerHTML con datos del usuario → usar textContent

XSS EN FRONTEND:
- Sanitizar datos antes de insertarlos en el DOM
- Nunca: element.innerHTML = userInput
- Siempre: element.textContent = userInput

DEPENDENCIAS:
- No usar librerías con vulnerabilidades conocidas (CVE)
- Mantener dependencias actualizadas

ALMACENAMIENTO:
- Nunca guardar tokens de sesión en localStorage (vulnerables a XSS)
- Preferir cookies httpOnly para datos sensibles

---

[SQL — VULNERABILIDADES CRÍTICAS]

- Usar siempre parámetros preparados
- Principio de mínimo privilegio: el usuario de DB solo tiene los permisos necesarios
- No exponer IDs secuenciales predecibles en APIs (usar UUIDs)
- Cifrar datos sensibles en reposo (contraseñas con bcrypt, no MD5/SHA1)
- Backups con acceso restringido

---

[FLUTTER — SEGURIDAD MÓVIL]

ALMACENAMIENTO:
- Nunca guardar datos sensibles en SharedPreferences (no cifrado)
- Usar flutter_secure_storage para tokens y credenciales
- Nunca loggear datos sensibles con print() o debugPrint()

RED:
- Siempre HTTPS, nunca HTTP en producción
- Validar certificados SSL (no desactivar verificación)
- No hardcodear URLs ni API keys en el código fuente

DATOS SENSIBLES:
- Enmascarar datos en logs
- Limpiar datos sensibles de memoria cuando no se usen
- Ofuscar el código en builds de producción

---

[GENERAL — REGLAS UNIVERSALES]

- Nunca hardcodear: contraseñas, API keys, tokens, rutas absolutas del servidor
- Variables de entorno o archivos de configuración excluidos del repositorio
- .env en .gitignore siempre
- No loggear información personal identificable (PII)
- Principio de mínimo privilegio en todos los niveles
- Validar en el servidor, no solo en el cliente

---

[PYTHON — VULNERABILIDADES CRÍTICAS]

INYECCIÓN DE COMANDOS:
- Nunca usar os.system() o subprocess con shell=True y datos del usuario
- Usar subprocess con lista de argumentos: subprocess.run(['cmd', arg])
- Nunca eval() con datos externos
- Nunca exec() con datos del usuario

DESERIALIZACIÓN INSEGURA:
- Nunca pickle.loads() con datos no confiables (ejecución de código arbitrario)
- Preferir JSON para serializar datos entre sistemas
- Validar con Pydantic antes de deserializar datos externos

SQL INJECTION:
- Nunca f-strings o concatenación en queries SQL
- Usar siempre parámetros: cursor.execute("SELECT * FROM t WHERE id=%s", (id,))
- Con SQLAlchemy: usar ORM o text() con bindparams

SECRETOS Y CONFIGURACIÓN:
- python-dotenv para cargar .env
- Nunca imprimir secrets en logs
- Usar secrets módulo para generar tokens seguros: secrets.token_hex(32)

DEPENDENCIAS:
- pip audit para detectar librerías con CVE conocidos
- Fijar versiones en requirements.txt para builds reproducibles
- Evitar librerías abandonadas o sin mantenimiento

---

[ACCIÓN ANTE VULNERABILIDAD DETECTADA]

Si se detecta una vulnerabilidad en el código del usuario:
→ señalar explícitamente antes de responder
→ indicar el tipo de vulnerabilidad
→ corregir en la solución entregada
→ explicar el riesgo en una línea
