[REGLAS DE TESTING]

El sistema aplica estas reglas cuando genera código o cuando el usuario
pide tests, o cuando el código generado debería incluir tests.

---

[PRINCIPIOS BASE]

- Testear comportamiento, no implementación interna
- Un test = un comportamiento verificado
- Tests deben ser independientes entre sí (sin orden requerido)
- Tests deben ser determinísticos (mismo resultado siempre)
- Nombres descriptivos: test_should_return_null_when_user_not_found
- AAA Pattern: Arrange → Act → Assert

---

[PIRÁMIDE DE TESTS]

UNITARIOS (base — más cantidad):
- Testean una función/clase en aislamiento
- Mockear dependencias externas (DB, APIs, email)
- Rápidos de ejecutar (ms por test)
- Sin efectos secundarios

DE INTEGRACIÓN (medio):
- Testean la interacción entre componentes reales
- Pueden usar DB de test real
- Verifican que los módulos trabajen juntos correctamente

E2E / FUNCIONALES (cima — menos cantidad):
- Testean flujos completos del usuario
- Lentos pero de alto valor para flujos críticos
- Solo para casos de uso principales

---

[PHP — TESTING]

Framework: PHPUnit
Convenciones:
```php
class UserServiceTest extends TestCase
{
    private UserService $service;
    private UserRepository $repository;

    protected function setUp(): void
    {
        $this->repository = $this->createMock(UserRepository::class);
        $this->service = new UserService($this->repository);
    }

    public function test_should_return_user_when_id_exists(): void
    {
        // Arrange
        $expected = new User(id: 1, name: 'Test');
        $this->repository
            ->expects($this->once())
            ->method('findById')
            ->with(1)
            ->willReturn($expected);

        // Act
        $result = $this->service->getUser(1);

        // Assert
        $this->assertEquals($expected, $result);
    }

    public function test_should_throw_when_user_not_found(): void
    {
        $this->expectException(UserNotFoundException::class);
        $this->repository->method('findById')->willReturn(null);
        $this->service->getUser(999);
    }
}
```

Reglas:
- Método de test siempre empieza con test_
- Un assert de comportamiento por test (puede haber múltiples asserts técnicos)
- setUp() para inicializar mocks, no en el constructor
- No lógica condicional en tests

[JAVASCRIPT — TESTING]

Framework: Vitest (preferido) / Jest
```javascript
// user.service.test.js
import { describe, it, expect, vi, beforeEach } from 'vitest'
import { UserService } from './user.service.js'

describe('UserService', () => {
    let service
    let repository

    beforeEach(() => {
        repository = { findById: vi.fn() }
        service = new UserService(repository)
    })

    it('should return user when id exists', async () => {
        // Arrange
        const expected = { id: 1, name: 'Test' }
        repository.findById.mockResolvedValue(expected)

        // Act
        const result = await service.getUser(1)

        // Assert
        expect(result).toEqual(expected)
        expect(repository.findById).toHaveBeenCalledWith(1)
    })

    it('should throw UserNotFoundError when user does not exist', async () => {
        repository.findById.mockResolvedValue(null)
        await expect(service.getUser(999)).rejects.toThrow('UserNotFoundError')
    })
})
```

[PYTHON — TESTING]

Framework: pytest

---

[QUÉ SIEMPRE TESTEAR]

- Happy path: el caso de uso normal y exitoso
- Not found / null: cuando el recurso no existe
- Validación: inputs inválidos (vacío, tipo incorrecto, fuera de rango)
- Error handling: comportamiento cuando una dependencia falla
- Límites: valores en el borde del rango válido

---

[QUÉ NO TESTEAR]

- Implementación interna (qué método interno se llamó, salvo contratos)
- Librerías de terceros (se asume que están testeadas)
- Getters/setters triviales sin lógica
- Código de configuración de frameworks

---

[CUÁNDO SUGERIR TESTS]

El sistema sugiere incluir tests cuando:
- El usuario pide crear un Service, Repository o UseCase
- El usuario pide debuggear un error que pudo haberse detectado con tests
- El usuario está en modo REFACTOR (tests de regresión son críticos)
- La solución tiene lógica condicional compleja

Formato de sugerencia:
→ Entregar código principal completo
→ Agregar sección Tests con los casos más críticos
→ No incluir tests para código trivial sin lógica

---

[FLUTTER / DART — TESTING]

Framework: flutter_test (built-in) + mockito
Widget Test

Reglas:
- Usar `setUp()` para inicializar mocks, nunca en el scope global
- `group()` para agrupar tests relacionados por método o caso de uso
- Usar `@GenerateMocks` de mockito para generar mocks (correr `dart run build_runner build`)
- Widget tests con `tester.pumpWidget()` y `tester.pump()` para ciclos de UI
- No testear widgets de UI trivial (solo si tienen lógica condicional)
- Cancelar timers y subscriptions en `tearDown()` si aplica

---

[COBERTURA ORIENTATIVA]

- Lógica de dominio (servicios, casos de uso): 80%+
- Repositorios: tests de integración
- Controllers/endpoints: test de integración (happy + error path)
- Utilidades con lógica: 90%+
- UI/presentación: opcional salvo lógica compleja
- Flutter widgets: widget tests solo para widgets con lógica (formularios, condicionales)
